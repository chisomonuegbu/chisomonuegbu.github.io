# Migrating Vaultwarden to Keycloak SSO: A Deep Dive

*A technical account of migrating a self-hosted password manager from stock Vaultwarden to OIDCWarden with Keycloak SSO — what went smoothly, what didn't, and what the problems revealed about the underlying systems.*

---

## Background

I run a self-hosted password manager on a VPS behind nginx, with Keycloak on a separate VPS behind Caddy. The goal was to connect the two: replace the stock `vaultwarden/server` image with `timshel/oidcwarden` and federate authentication through Keycloak using OpenID Connect.

This isn't a setup guide. It's an account of the decisions made, the problems encountered, and what each one revealed.

---

## 1. Choosing OIDCWarden Over Stock Vaultwarden

The first decision was architectural: how to add SSO to an existing Vaultwarden instance.

Stock Vaultwarden has basic OIDC support, but it was added incrementally and has known rough edges around session management. OIDCWarden is a maintained fork by [Timshel](https://github.com/Timshel/OIDCWarden) that treats SSO as a first-class feature rather than a bolt-on. The specific capability that mattered was the `sso_users` table.

In stock Vaultwarden, the SSO identifier is stored in a column on the `users` table (`external_id`). In OIDCWarden, it lives in a dedicated `sso_users` table as a composite of the `{iss}/{sub}` claims from the OIDC ID token — the issuer URL and the user's subject identifier from the identity provider.

This design decision has real consequences. The identifier in `sso_users` looks like:

```
https://auth.example.com/realms/myrealm/00000000-0000-0000-0000-000000000001
```

That's `{iss}/{sub}` — the full Keycloak realm URL combined with the Keycloak user UUID. Separating this from the vault user UUID means:

- You can change your SSO provider without orphaning vault accounts
- The vault user UUID stays stable across IdP migrations
- Account takeover via email change is prevented — the binding is to the IdP identity, not just the email address

The migration itself was straightforward at the image level. The `vaultwarden-data` volume is fully compatible — OIDCWarden only adds the `sso_users` table and an `external_id` column to the `organizations` table via its built-in migration system. No manual schema work required.

The Docker Compose change was a one-line image swap plus a block of SSO environment variables:

```yaml
image: timshel/oidcwarden:latest

# SSO additions
- SSO_ENABLED=true
- SSO_ONLY=false  # keep false until SSO is confirmed working
- SSO_AUTHORITY=https://${KEYCLOAK_FQDN}/realms/${KEYCLOAK_REALM}
- SSO_CLIENT_ID=${SSO_CLIENT_ID}
- SSO_CLIENT_SECRET=${SSO_CLIENT_SECRET}
- SSO_PKCE=true
- SSO_SCOPES=profile email offline_access
- SSO_FRONTEND=button
```

## 2. The SQLite Snapshot Timing Problem

Before touching anything, I took a backup of the Vaultwarden data directory. This revealed something worth understanding about SQLite's WAL mode.

SQLite in WAL (Write-Ahead Logging) mode splits the database across three files:

- `db.sqlite3` — the main database file
- `db.sqlite3-wal` — the WAL file containing recent uncommitted transactions
- `db.sqlite3-shm` — the shared memory index for the WAL

When you stop a container that's using a SQLite database, the database connection closes and SQLite performs a **checkpoint** — it merges the WAL transactions back into the main database file and deletes the WAL and SHM files. The result is a single clean `db.sqlite3`.

My first backup was taken using `cp -a` shortly after stopping the container. The container stop returned in about a second, and I ran the copy a few minutes later. The backup captured this:

```
db.sqlite3       292K  (timestamped — not yet updated)
db.sqlite3-wal   2.7M  (timestamped after the stop — active WAL)
db.sqlite3-shm    32K  (timestamped after the stop)
```

The WAL file was written *after* the stop command — which means the checkpoint was in progress when I copied the directory. SQLite was actively flushing data into the WAL as part of its shutdown sequence, and the copy captured it mid-flight.

A second backup, taken later after a clean stop, showed:

```
db.sqlite3       368K  (fully checkpointed, no WAL)
```

The 368K is larger than 292K because it now contains everything that was previously sitting in the WAL. Both backups are restorable — a WAL-mode backup with both the DB and WAL files present is valid — but the second is cleaner.

The lesson: `sqlite3 .backup` is more reliable than `cp -a` for live or recently-stopped databases because it uses SQLite's internal backup API, which handles WAL state correctly regardless of checkpoint status. If using `cp -a`, add a brief pause after the container stop:

```bash
docker compose stop vaultwarden
sleep 5
cp -a /path/to/vaultwarden-data /path/to/backup-$(date +%Y%m%d_%H%M%S)
```

## 3. Keycloak Client Configuration — The Non-Obvious Settings

Keycloak client configuration has a lot of fields. Most have sensible defaults. A few don't, and the ones that matter for a password manager are worth calling out explicitly.

### The Access Token Lifetime Collision

Keycloak's default access token lifetime is 5 minutes. The Bitwarden client has a hardcoded threshold: if the access token lifetime is 5 minutes or less, the client considers it already expired on receipt and immediately fires a refresh request. Since the freshly issued token is already at or past the threshold by the time the refresh fires, you get a race condition:

```
Token issued (t=0)
Client detects token expiring → fires refresh at ~t=4:30
Keycloak considers token expired at t=5:00
Refresh response arrives with a token that's already expired
→ Client treats this as auth failure → session dropped → logout
```

The fix is to raise the access token lifetime above 5 minutes at the realm or client level. A 15-minute lifetime gives a 10-minute window for the silent refresh to complete — more than sufficient even on a slow connection.

This is documented in the OIDCWarden repository but easy to miss. The symptom — being randomly logged out of the vault — looks identical to a vault lock timeout, which makes it hard to diagnose without knowing what to look for.

### Requiring PKCE at the Client Level

OIDCWarden is configured with `SSO_PKCE=true`, which means it will always include a PKCE code challenge in the authorization request. Keycloak has a separate client-level setting: **Require PKCE**. Leaving this off means Keycloak will accept auth requests with or without PKCE — the protection is optional rather than enforced.

The correct posture is to turn **Require PKCE** on and set the challenge method to **S256**. This ensures the auth flow cannot silently degrade to non-PKCE if something is misconfigured on the OIDCWarden side. For a password manager, you want the security boundary enforced at both ends.

**Plain** PKCE is not an acceptable challenge method — it provides no real security improvement over no PKCE. S256 only.

### The Implicit Flow Setting

Keycloak shows an **Access Token Lifespan for Implicit Flow** setting alongside the regular access token lifetime. This is irrelevant for OIDCWarden. Implicit flow is a deprecated OAuth pattern that was superseded by Authorization Code flow with PKCE — exactly what OIDCWarden uses. The setting exists for backward compatibility and can be left at its default.

### Session Timeouts

The default SSO Session Max of 10 hours creates a mismatch with a 7-day idle setting — the hard max cuts in first, forcing a full re-authentication after 10 hours regardless of activity. A reasonable set of values:

| Setting | Value | Reasoning |
|---|---|---|
| Access Token Lifespan | 15 min | Avoids the 5-minute race condition |
| SSO Session Idle | 7 days | Inactive sessions expire after a week |
| SSO Session Max | 30 days | Hard ceiling regardless of activity |

## 4. The CSP Header Surgery

The nginx reverse proxy configuration required no structural changes — OIDCWarden listens on the same port and handles the OIDC callback at `/identity/connect/oidc-signin`, which falls under the existing `/identity` location block.

What did require changes was the Content Security Policy header. The existing CSP was:

```nginx
add_header Content-Security-Policy "default-src 'self'; 
  script-src 'self' 'wasm-unsafe-eval'; 
  style-src 'self' 'unsafe-inline'; 
  img-src 'self' data: https://haveibeenpwned.com; 
  connect-src 'self' wss://$host https://api.pwnedpasswords.com; 
  frame-ancestors 'self'; 
  form-action 'self'; 
  base-uri 'self'" always;
```

The OIDC redirect flow requires the browser to make requests to the Keycloak server directly — not through nginx, but directly from the browser to the IdP. A restrictive CSP blocks these at the browser level silently, causing the SSO flow to fail with no obvious error in nginx logs.

Three directives needed updating:

**`connect-src`** — the browser makes fetch/XHR requests to Keycloak during the token exchange. Without the Keycloak FQDN here, these are blocked.

**`frame-src`** — Keycloak's login page renders in a redirect context that some browsers treat as a frame. The Keycloak origin needs to be explicitly permitted.

**`form-action`** — Keycloak's login form POSTs back to the callback URL. The destination needs to be whitelisted here as well.

The updated directive:

```nginx
add_header Content-Security-Policy "default-src 'self'; 
  script-src 'self' 'wasm-unsafe-eval'; 
  style-src 'self' 'unsafe-inline'; 
  img-src 'self' data: https://haveibeenpwned.com; 
  connect-src 'self' wss://$host https://api.pwnedpasswords.com https://your-keycloak-fqdn; 
  frame-src https://your-keycloak-fqdn; 
  frame-ancestors 'self'; 
  form-action 'self' https://your-keycloak-fqdn; 
  base-uri 'self'" always;
```

This is not documented in any of the OIDCWarden, Vaultwarden, or Keycloak setup guides. It's the kind of thing that only surfaces when you have a hardened nginx config. The symptom — Keycloak login page renders correctly but the redirect back to the vault silently fails — is easy to misattribute to a Keycloak or OIDCWarden misconfiguration.

## 5. The Keycloak Realm Data Loss Investigation

After completing the Keycloak client configuration and switching the OIDCWarden image, the first login attempt returned `client_not_found`. The error came from Keycloak itself, not OIDCWarden — Keycloak received the auth request, found the realm, but found no matching client.

The immediate suspicion was a persistence problem. Keycloak's default development mode uses an embedded H2 in-memory database that wipes on restart. If the production setup had fallen back to this for any reason, all realm configuration would be lost on container restart.

Checking the compose file confirmed this wasn't the case — the setup used PostgreSQL with a bind-mounted volume. The postgres data directory had content. Persistence was configured correctly.

A direct database query confirmed the realm was intact:

```sql
SELECT id, name FROM realm;

                  id                  |       name
--------------------------------------+-------------------
 00000000-0000-0000-0000-00000000aaaa | master
 00000000-0000-0000-0000-00000000bbbb | myrealm
```

Both realms present. The Keycloak logs then showed the actual error:

```json
"type": "LOGIN_ERROR",
"realmName": "myrealm",
"clientId": "vaultwarden-sso",
"error": "client_not_found"
```

The realm existed. The client did not. The most likely explanation: the client was configured in the Keycloak UI but the browser session expired or navigation occurred before the final **Save** was committed. Keycloak's admin console is a single-page app — partial configuration is held in the browser until saved, and an expired session or accidental navigation discards it without warning.

The fix was to recreate the client from scratch. All realm-level settings (token lifetimes, session timeouts, PKCE requirements) were preserved — only the client record needed to be rebuilt.

The investigation was a useful reminder that "data loss" and "configuration never saved" look identical from the outside.

## 6. The `sso_users` Table Conflict

After the initial setup was working, I migrated to a new Keycloak instance. The redirect flow worked — Keycloak login page appeared, credentials accepted, redirect back to OIDCWarden — but the vault returned:

```
SSO client with the same email already exists
```

This is a safety guard, not a bug. On first SSO login, OIDCWarden writes a row to the `sso_users` table:

```
user_uuid                              | external_id                                                                       | created_at
---------------------------------------+-----------------------------------------------------------------------------------+------------
00000000-0000-0000-0000-000000000001   | https://old-keycloak.example.com/realms/myrealm/00000000-0000-0000-0000-0000abcd | 2026-04-15
```

The `external_id` is the `{iss}/{sub}` composite — the issuer URL from the old Keycloak instance combined with the Keycloak user's subject UUID. When the new Keycloak instance issued a token, it presented a different `iss` value. OIDCWarden found an existing `sso_users` row for that vault account, saw the issuer mismatch, and blocked the re-link.

This protection exists for good reason — if OIDCWarden silently reassigned accounts to any new IdP claiming the same email, a misconfigured or malicious provider could hijack existing vault accounts.

The fix is documented in the OIDCWarden SSO.md:

> If you ever need to change the provider URL or the provider itself, you'll have to first delete the association, then ensure that `SSO_SIGNUPS_MATCH_EMAIL` is set to true so the account can be re-linked on next login.

My initial instinct was to null the `external_id` column on the `users` table — this was wrong. That column is a legacy field. The actual binding lives in `sso_users`. Checking the OIDCWarden documentation before making the change avoided an incorrect database modification.

With the container stopped:

```sql
-- Confirm the binding
SELECT * FROM sso_users WHERE user_uuid = '<your-vault-user-uuid>';

-- Delete it
DELETE FROM sso_users WHERE user_uuid = '<your-vault-user-uuid>';
```

On next login, `SSO_SIGNUPS_MATCH_EMAIL=true` (the default) caused OIDCWarden to match the new Keycloak identity to the existing vault account by email and write a fresh `sso_users` row with the new issuer URL.

## 7. The Two-Layer Session Model

During testing, the vault occasionally returned to an unlock screen requesting the master password. This initially looked like a session management problem — either a Keycloak timeout or an OIDCWarden bug.

It isn't one problem. It's two independent session layers that happen to produce the same symptom.

### Layer 1 — The Keycloak SSO Session

This layer is controlled by the Keycloak settings configured earlier (15-minute access token, 7-day idle, 30-day max). It governs whether Keycloak considers the user authenticated.

Within this layer, there are two tokens:

**Access token** — short-lived (15 minutes), sent with every API call. It's a JWT — cryptographically signed and self-contained. The server validates it without a database lookup, which is why it scales well but also why it cannot be revoked before expiry. This is the fundamental tradeoff of stateless token validation.

**Refresh token** — long-lived (7-day idle / 30-day max), rarely sent over the wire. Its only job is to obtain a new access token when the current one expires. It IS tracked server-side, which is why it can be revoked instantly (logout, session termination, account deletion).

The Bitwarden client has a hardcoded 5-minute early-refresh trigger — when the access token has less than 5 minutes remaining, the client fires a silent background refresh. With a 15-minute token lifetime, this silent refresh happens approximately every 10 minutes of active use, completely transparently.

If the client goes offline for more than 15 minutes, the access token expires and cannot be silently refreshed. The session drops. This is expected behaviour. When the client comes back online, it presents the refresh token to recover the session — as long as it hasn't been idle for more than 7 days or exceeded the 30-day hard maximum.

### Layer 2 — The Bitwarden Client Vault Lock

This layer is entirely local. It has nothing to do with Keycloak. The Bitwarden client maintains an in-memory decryption key for the vault. When the vault lock timeout triggers — configurable in client settings under **Settings → Security → Vault timeout** — the client clears that key from memory and returns to the unlock screen.

This is a deliberate local security boundary. The master password is what decrypts the vault data — Keycloak only proves identity, it doesn't handle vault decryption. These two concerns are intentionally separated by design.

### Why They Look Identical

Both layers return you to an unlock/login screen. The difference:

| Trigger | What you see | What happened |
|---|---|---|
| Vault lock timeout (Layer 2) | Master password prompt | Local key cleared, no network involved |
| Access token expired (Layer 1) | Redirect to Keycloak login | Session dropped, full re-auth required |
| Refresh token expired (Layer 1) | Redirect to Keycloak login | Long-term session ended, full re-auth required |
| Browser cache cleared | Redirect to Keycloak login | Refresh token destroyed, Keycloak has no record of it |

The vault lock timeout is the most common — set it to something sensible in the client settings and it stops being surprising. The access token expiry is prevented by a sufficient lifetime configuration. The refresh token scenarios are rare by design.

## Takeaways

A few things this migration reinforced:

**Read the source documentation before touching the database.** My first instinct for the `sso_users` conflict was to null the `external_id` column — which would have done nothing because that's not where OIDCWarden stores SSO bindings. One grep through the OIDCWarden SSO.md clarified the correct table and the correct approach.

**CSP failures are silent.** A browser blocked by CSP produces no server-side log entries. The nginx access log looks clean. The error only appears in the browser's developer console, and it manifests as the SSO flow silently failing rather than a visible error message. When integrating a cross-origin auth flow into a hardened nginx config, the CSP is always the first place to check.

**SQLite WAL mode is not a detail.** Understanding that a stopped container may not have completed its checkpoint before you copy the data directory is the difference between a reliable backup strategy and one that occasionally captures an inconsistent state.

**The two-token model is universal.** Access token + refresh token is the same pattern across AWS STS, Google OAuth2, GitHub App tokens, Kubernetes service account JWTs, and virtually every other secured API. The lifetimes and storage strategies vary, but the underlying tradeoff — stateless short-lived credentials for API calls, backed by revocable long-lived credentials for renewal — is consistent. Understanding it once transfers everywhere.
